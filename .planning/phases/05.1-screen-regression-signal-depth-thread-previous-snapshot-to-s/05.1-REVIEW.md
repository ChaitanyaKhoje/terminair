## Summary

Phase 5.1 added `get_previous_models()` to `StateAggregator` and `MockDataProvider`, then wired `self._previous_models` through `ModelListScreen`, `ProblemsScreen`, and `ModelDetailScreen` so grain/upstream signals are reachable. The implementation is broadly correct: both providers return a proper `list[ModelState]`, screens initialize `_previous_models = []` in `__init__` before any `_render()` could fire, and the `on_mount` clock fix correctly calls `_update_header()` once synchronously.

Five warnings and three informational issues were found. The most impactful is that `ModelDetailScreen._render_regression()` silently drops `upstream_schema_change` signals because it only passes one model to `RegressionAnalyzer`. The second is a hollow TDD test that documents intent but asserts nothing about the property it was written to verify.

## Warnings

### WR-01: `_render_regression` in `detail.py` silently drops `upstream_schema_change` signals

**File:** `terminair/screens/detail.py:163-165`

**Issue:** `_render_regression` constructs a `RegressionAnalyzer` with only the single current model: `RegressionAnalyzer([model])`. Inside `RegressionAnalyzer.analyze()`, the `upstream_schema_change` detector iterates `model.upstream_deps` and looks each dep up in `self._current`. Because `self._current` contains only the one model, no upstream dep is ever found, so `upstream_schema_change` signals are never emitted in the detail view — even when they would appear on the problems screen for the same model.

```python
# current (silently drops upstream_schema_change):
signals = RegressionAnalyzer([model]).analyze(previous=previous)

# fix — pass the full model list so upstream deps are resolvable:
signals = RegressionAnalyzer(self._models).analyze(previous=self._previous_models or None)
# then filter to only this model's signals:
signals = [s for s in signals if s.node_id == model.node_id]
```

### WR-03: `MockDataProvider.get_previous_models()` copies current model state, not a true snapshot

**File:** `terminair/dbt/mock_data.py:282-305`

**Issue:** `get_previous_models()` reads `current = list(self._models)` at call time. Because `self._models` is mutated by `tick()` (status, duration_s, rows_written, row_delta_pct), calling `get_previous_models()` after a tick returns models with the post-tick state as the "previous" snapshot. For the grain-drift use case this is intentional, but for a caller that calls `get_models()` and then `get_previous_models()` in sequence (as all three screens do), the two snapshots are always coherent. The subtle risk is that `tick()` mutates model objects in-place (e.g. `m.status = "success"`) and those same objects are referenced by the `previous` list because the shallow copy via `list(self._models)` still holds the same `ModelState` instances. If a screen holds a reference to the returned previous list after a tick, it will silently observe the post-tick state.

```python
# current — shallow copy of list, same ModelState objects:
current = list(self._models)

# fix — deep-copy each ModelState to produce a true snapshot:
import copy
current = [copy.copy(m) for m in self._models]
```

### WR-04: `_update_statusbar` in `model_list.py` calls `RegressionAnalyzer` redundantly after `_render()` already ran it

**File:** `terminair/screens/model_list.py:180-189`

**Issue:** `_render()` calls `_update_statusbar()` (line 147), which itself calls `RegressionAnalyzer(self._models).analyze(...)`. A second identical analysis run is never cached. The `ProblemsScreen` avoids this by storing `self._signals` from its single `analyze()` call; `ModelListScreen` discards the signals from `_render` and recomputes them inside `_update_statusbar`. This is a correctness concern only if `_update_statusbar` is ever called outside of `_render()` (it is not currently), but the redundancy could cause the statusbar count to diverge from the table if signals computation were ever refactored.

```python
# fix — compute signals once in _render() and pass them in:
def _render(self) -> None:
    ...
    signals = RegressionAnalyzer(self._models).analyze(previous=self._previous_models or None)
    ...
    self._update_statusbar(signals)

def _update_statusbar(self, signals: list) -> None:
    warnings = sum(1 for s in signals if s.severity in ("critical", "warning"))
    ...
```

### WR-05: `action_refresh` in `base.py` calls base `_load_models()` — but due to Python's dynamic dispatch it calls the override, making the base implementation unreachable dead code

**File:** `terminair/screens/base.py:43-51`

**Issue:** `_queue_reload` calls `self._load_models()`. Python resolves this via `self` at runtime, so the overridden `_load_models()` on each subclass is called correctly. However, the base class provides a concrete implementation of `_load_models()` (lines 43-48) that will never execute for any screen that overrides it — and every screen that added `get_previous_models()` fetching (`ModelListScreen`, `ProblemsScreen`, `ModelDetailScreen`) overrides it. The base implementation skips `get_previous_models()`, so any screen that does NOT override `_load_models()` (future screens, or a screen added by mistake without the override) will miss populating `_previous_models` and regress silently to no grain signals.

This is not a bug in the current code, but the base `_load_models` is now a footgun: the correct pattern is to override it, but there is nothing that forces subclasses to do so.

```python
# fix — make the base implementation raise NotImplementedError like _render():
async def _load_models(self) -> None:
    raise NotImplementedError
```

Or, lift the common pattern into the base class:

```python
async def _load_models(self) -> None:
    provider = self.app_typed.get_data_provider()
    self._models = list(await provider.get_models())
    self._previous_models = await provider.get_previous_models()
    self._sync_selected_model()
    self._render()
```

and add `self._previous_models: list[ModelState] = []` to `DbtScreen.__init__`.

### IN-01: `get_previous_models` in `aggregator.py` iterates all manifest nodes even when `_prev_index` is always empty

**File:** `terminair/dbt/aggregator.py:226-253`

**Issue:** When `ArtifactReader` is initialized without `previous_path`, `_prev_index` is `None` and `get_previous_result()` always returns `None`. `get_previous_models()` iterates all manifest nodes, calls `get_previous_result()` for each (always `None`), skips all of them (line 228-229), and returns `[]`. This is correct but wasteful. A cheap short-circuit can make the intent explicit and avoid the O(n) traversal:

```python
async def get_previous_models(self) -> list[ModelState]:
    if self._artifacts._prev_index is None:
        return []
    ...
```

This requires either accessing a private attribute or adding `ArtifactReader.has_previous_results() -> bool`. The latter is cleaner.

### IN-02: `stg_orders` mock row_delta_pct comment is misleading

**File:** `terminair/dbt/mock_data.py:154`

**Issue:** The inline comment says `# row_drop WARNING` but the threshold for WARNING is `< -10%`. The value `-25.0` is between `-10%` and `-30%` and does generate a WARNING-level signal. However, the `_build_initial_models` docstring (line 14) says "Two models have row_delta_pct < -25% (row_drop signals)." That's inaccurate: `stg_orders` has exactly `-25.0%`, not `< -25%`. `mart_platform_usage` has `-40.67%` which is below `-30%` (CRITICAL). The comment and docstring describe different thresholds. The test `test_row_drop_signals_present` asserts `>= 2` and passes because both models produce a signal — but the docstring misleads future readers about which threshold applies.

```python
# fix — correct the docstring to match actual signal thresholds:
# Two models trigger row_drop signals: stg_orders (-25.0% → WARNING ≥-10%),
# mart_platform_usage (-40.67% → CRITICAL ≥-30%).
```
