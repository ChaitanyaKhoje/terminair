# Phase 5.1: Screen Regression Signal Depth - Research

**Researched:** 2026-05-16
**Domain:** Textual screen lifecycle, RegressionAnalyzer wiring, data provider interface extension
**Confidence:** HIGH

## Summary

This phase resolves two specific tech debt items found during the v1.0 audit. Both fixes are surgical and self-contained — no new dependencies, no architectural changes.

**TD-01** (`grain_added`, `grain_removed`, `upstream_schema_change` signals unreachable): The `RegressionAnalyzer.analyze(previous=...)` parameter already exists and works correctly. The gap is purely in wiring: `StateAggregator` and `MockDataProvider` have no `get_previous_models()` method, so screens call `analyze()` without a previous snapshot and those three signal types are permanently silent. The fix adds `get_previous_models()` to both providers and updates the three call sites (`ModelListScreen._update_statusbar()`, `ProblemsScreen._render()`, `ModelDetailScreen._render_regression()`).

**TD-02** (clock never fires on initial mount): `ModelListScreen` starts the 1-second clock interval only in `on_screen_resume`. On initial TUI launch the app calls `_switch_to("model_list")` → `push_screen("model_list")`, which triggers `on_mount` but not `on_screen_resume` (Textual fires `on_screen_resume` only when a screen already in the stack is *revealed* after being covered, not on first push). The fix: call `_update_header()` once directly inside `on_mount`.

**Primary recommendation:** Add `get_previous_models() -> list[ModelState]` to both `StateAggregator` and `MockDataProvider`; pass the result to `analyze()` at each of the three call sites. Call `_update_header()` once in `ModelListScreen.on_mount()`.

## Architecture Patterns

### System Architecture Diagram

```

          │
          ▼
   ArtifactReader._prev_index   ← already loaded from previous run_results.json
          │
          ▼
   StateAggregator.get_previous_models()   ← NEW method
          │                  │

          │                     from _prev_index (same logic as get_models()
          │                     but using get_previous_result() per node)
          │
          ├──► ModelListScreen._update_statusbar()
          │          └─ RegressionAnalyzer(current).analyze(previous=prev)
          │
          ├──► ProblemsScreen._render()
          │          └─ RegressionAnalyzer(current).analyze(previous=prev)
          │
          └──► ModelDetailScreen._render_regression(model)
                     └─ RegressionAnalyzer([model]).analyze(previous=[prev_model])
                                                                 (looked up by node_id)
```

For `MockDataProvider`, the "previous snapshot" is the same model list but with slightly shifted row counts — enough to exercise `grain_added`/`grain_removed`/`upstream_schema_change` detection in tests.

### Pattern 1: `get_previous_models()` — separate method (recommended over tuple return)

**What:** Add `async def get_previous_models() -> list[ModelState]` to both providers.

**Why separate method instead of `(current, previous)` tuple:**
- Preserves existing call sites in `DbtScreen._load_models()` which call `await provider.get_models()` — no churn to base class
- Screens that don't need previous snapshot (e.g., `LineageScreen`) are unaffected
- Screens that do need it call `get_previous_models()` in their own `_load_models` override or in `_render()`
- Simpler to test each method independently

**Call pattern in screens (recommended):**

```python
# [VERIFIED: direct codebase reading]
# In DbtScreen._load_models() — base class, unchanged
async def _load_models(self) -> None:
    provider = self.app_typed.get_data_provider()
    models = await provider.get_models()
    self._models = list(models)
    self._sync_selected_model()
    self._render()
```

Screens that need previous snapshot should fetch it lazily inside `_render()` via a stored provider reference, or override `_load_models()` to also fetch `_previous_models`. The simplest approach: store `self._previous_models: list[ModelState] = []` on each affected screen and populate it in an overridden `_load_models()`.

**Example override in ProblemsScreen / ModelListScreen:**

```python
# [VERIFIED: direct codebase reading — pattern derived from existing _load_models]
async def _load_models(self) -> None:
    provider = self.app_typed.get_data_provider()
    models = await provider.get_models()
    self._models = list(models)
    self._previous_models = await provider.get_previous_models()
    self._sync_selected_model()
    self._render()
```

### Pattern 2: `StateAggregator.get_previous_models()` implementation

**What:** Re-run the ModelState assembly loop over `_artifacts._prev_index`. The previous snapshot does not need Airflow or Snowflake enrichment — only grain_columns (from manifest) and materialization are needed for grain/schema-change signals.

**Minimal implementation — only fields used by RegressionAnalyzer:**

```python
# [VERIFIED: reading regression.py — only these fields are accessed on prev_map items]
# grain_columns  → for grain_added / grain_removed
# materialization → for upstream_schema_change
# node_id        → used as dict key in prev_map
```

A full `ModelState` can be constructed from manifest data alone (grain_columns, materialization come from the manifest, which is always current). `rows_previous` and `rows_written` on the previous snapshot are not used by RegressionAnalyzer — they are only consulted on the current snapshot to compute `row_delta_pct`.

**Simplest correct approach:** Build minimal `ModelState` instances from manifest data for each node present in `_artifacts._prev_index`. All fields not needed for regression can default to empty/None.

### Pattern 3: `MockDataProvider.get_previous_models()` for demo/test

**What:** Return a variant of the 10 models with grain_columns tweaked to produce at least one `grain_added` and one `grain_removed` signal in demo mode.

**Example:**
- `fct_revenue_daily`: current `grain_columns=["revenue_date"]`; previous returns `grain_columns=["revenue_date", "region"]` → `grain_removed` signal (current is shorter)
- `fct_risk_exposure`: current `grain_columns=["risk_date", "entity_id"]`; previous returns `grain_columns=["risk_date"]` → `grain_added` signal (current is longer)

This gives visible demo coverage of all three newly-reachable signal types.

### Pattern 4: TD-02 — Clock on initial mount

**What:** Textual fires `on_screen_resume` only when a screen is *unpaused* (revealed after being covered), not on first push. `on_mount` fires on first push. [VERIFIED: direct codebase reading — `on_screen_resume` vs `on_mount` semantics]

**Current code:**
```python
# [VERIFIED: terminair/screens/model_list.py line 94-95]
def on_screen_resume(self) -> None:
    self._clock_timer = self.set_interval(1.0, self._update_header)
```

**Fix:** Add a direct call in `on_mount`:
```python
async def on_mount(self) -> None:
    self._update_header()        # fire once immediately
    await self._load_models()
```

This does NOT start a second interval — `on_screen_resume` still owns the repeating timer. The single direct call in `on_mount` fills the gap for the initial render. [ASSUMED: Textual does not fire on_screen_resume for the first push of a screen — this matches the observed bug and the CONTEXT.md description, but not independently verified against Textual changelog]

### Anti-Patterns to Avoid

- **Double interval:** Do NOT call `self.set_interval(1.0, self._update_header)` in `on_mount` — that starts a second 1-second timer, so the clock fires twice per second after the user navigates away and back. Call `_update_header()` once directly.
- **Tuple return from `get_models()`:** Returning `(current, previous)` would require touching `DbtScreen._load_models()` in the base class and every screen's call site. The separate `get_previous_models()` method is additive — zero breakage.
- **Calling `get_previous_models()` inside `_render()`:** `_render()` is synchronous. Fetching from the provider is async. Previous models must be fetched in `_load_models()` (async) and stored as instance state for `_render()` to read.
- **Building previous ModelState with full Airflow/Snowflake lookup:** Unnecessary. RegressionAnalyzer only reads `grain_columns`, `materialization`, and `node_id` from the previous snapshot.

## Common Pitfalls

### Pitfall 1: `on_screen_resume` not firing on initial push
**What goes wrong:** Developer adds clock-start logic only to `on_screen_resume` — clock never ticks until user leaves and returns to the screen.
**Why it happens:** Textual's `on_screen_resume` fires when a covered screen is revealed. The first `push_screen()` fires `on_mount`, not `on_screen_resume`.
**How to avoid:** Call `_update_header()` once in `on_mount` in addition to the repeating interval in `on_screen_resume`.
**Warning signs:** Clock reads stale/empty time on initial TUI launch; refreshes after pressing Esc.

### Pitfall 2: `get_previous_models()` called in synchronous `_render()`
**What goes wrong:** `_render()` is a plain `def`. Calling `await provider.get_previous_models()` there is a syntax error. Using `asyncio.run()` inside a Textual callback deadlocks.
**Why it happens:** Forgetting that `_render()` is sync while data fetching is async.
**How to avoid:** Fetch previous models in the overridden `_load_models()` (async) and store as `self._previous_models`. `_render()` reads `self._previous_models` without any async call.
**Warning signs:** `SyntaxError: 'await' outside function` or event-loop-blocked TUI on screen transitions.

### Pitfall 3: `MockDataProvider.get_previous_models()` returns the same list as `get_models()`
**What goes wrong:** `grain_added`/`grain_removed`/`upstream_schema_change` signals never fire in demo mode because current == previous.
**Why it happens:** Easy to return `self._models` unchanged as the "previous" snapshot.
**How to avoid:** Deliberately mutate grain_columns on a subset of models in the previous snapshot so the diff produces signals.
**Warning signs:** Demo mode shows 0 grain signals even after wiring.

### Pitfall 4: `StateAggregator.get_previous_models()` returns empty list when `previous_path` not set
**What goes wrong:** Screens pass an empty list to `analyze(previous=[])`, which `RegressionAnalyzer` treats the same as `previous=None` only if the caller checks length first — but actually, `prev_map = {m.node_id: m for m in previous}` with an empty list produces an empty dict, which is functionally identical to `previous=None`. This is correct behavior, not a bug.
**Why it happens:** No issue — empty previous list is safe. Document to avoid confusion.
**How to avoid:** No special-casing needed. Returning `[]` when previous_path not set is the correct contract.

## Code Examples

### TD-01: StateAggregator.get_previous_models() — minimal build

```python
# [VERIFIED: reading aggregator.py and regression.py to determine which fields matter]
async def get_previous_models(self) -> list[ModelState]:
    """Return previous-run ModelState list for RegressionAnalyzer.

    Only grain_columns and materialization are populated — sufficient for
    grain_added, grain_removed, and upstream_schema_change detection.
    Returns [] when run_results_previous_path is not configured.
    """
    models: list[ModelState] = []
    for node_id in self._manifest.get_all_node_ids():
        prev_result = self._artifacts.get_previous_result(node_id)
        if prev_result is None:
            continue  # model not in previous run — skip
        node = self._manifest.get_node(node_id)
        if node is None:
            continue
        name = node.get("name", node_id.split(".")[-1])
        config = self._manifest.get_config(node_id)
        materialization = config.get("materialized", "unknown")
        grain_columns = self._manifest.get_grain_columns(node_id)
        # Fill only fields RegressionAnalyzer reads from prev_map
        models.append(
            ModelState(
                node_id=node_id,
                name=name,
                tag="",
                status="",
                dag_id="",
                task_id="",
                materialization=materialization,
                schema_name="",
                database_name="",
                has_upstream_failure=False,
                grain_columns=grain_columns,
            )
        )
    return models
```

### TD-01: ModelDetailScreen — single-model previous lookup

```python
# [VERIFIED: reading screens/detail.py _render_regression()]
def _render_regression(self, model: ModelState) -> str:
    prev_model = next(
        (m for m in self._previous_models if m.node_id == model.node_id), None
    )
    previous = [prev_model] if prev_model is not None else None
    signals = RegressionAnalyzer([model]).analyze(previous=previous)
    if not signals:
        return "No regression signals for this model."
    lines = []
    for signal in signals:
        lines.append(f"{signal.severity.upper()}: {signal.signal_type} — {signal.description}")
    return "\n".join(lines)
```

## Open Questions

1. **Does `get_previous_models()` need to be `async`?**
   - What we know: `get_models()` is `async def` (verified). `StateAggregator.get_models()` is async because it may call `AirflowBridge.get_task_statuses()`. `get_previous_models()` has no awaitable calls in its implementation.
   - What's unclear: Should the interface match `get_models()` exactly (both async) for Liskov-substitution consistency?
   - Recommendation: Declare `async def get_previous_models()` on both providers to keep the interface symmetric, even though the implementation has no awaits. This lets screens call it with `await` uniformly.

2. **Should `ProblemsScreen` and `ModelListScreen` fetch previous models via an overridden `_load_models()` or by reading from `self.app_typed.get_data_provider()` directly in `_render()`?**
   - What we know: `_render()` is synchronous; provider calls are async. Cannot call provider directly in `_render()`.
   - Recommendation: Override `_load_models()` in each affected screen. This is the only safe pattern.
